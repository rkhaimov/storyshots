# Подмена зависимостей

После определения метода верификации, необходимо указать, какие зависимости стоит использовать при тестировании, а каких
стоит избегать.

## Исключение сервера

<small style={{ color: 'green' }}>📈 ⚡ Удовлетворяет</small>  
<small style={{ color: 'red' }}>🛡️ 🔧 Снижает</small>

Сервер это прежде всего внешний источник данных для приложения, но проблема не в этом, а в том что он является
*недетерминированным*.

:::note
Детерминированные (или же идемпотентные) функции - это такие функции возвращаемый результат которых полностью
предсказывается входящими в неё аргументами.
:::

Ввиду отсутствия прямого контроля, поведение таких систем не детерминировано: нет никакой гарантии, что она будет
функционировать на момент запуска теста. Даже в случае работоспособности, отсутствует гарантия прогнозируемого времени и
формы ответа.

```ts
fetch('/api/users') // []

// Спустя некоторое время работы с системой, тот же метод возвращает уже другой результат при тех же аргументах
fetch('/api/users') // [{ user: 'Vasiliy' }]
```

К тому же, сервер требует определенного времени на выполнение своих функций, это не только добавляет дополнительные
асинхронные компоненты в тесты, но и увеличивает время их выполнения.

:::note
Стоит понимать что тесты, не зависящие от backend не предоставляют такой же защиты от регресса что и, например, E2E
сценарии. Однако, это сделано в угоду повышенной поддерживаемости.
:::

## Исключение webapi*

<small style={{ color: 'green' }}>📈 ⚡ Удовлетворяет</small>  
<small style={{ color: 'red' }}>🛡️ 🔧 Снижает</small>

Backend не является единственным примером не идемпотентного источника данных, к таковым можно также отнести и некоторые
компоненты webapi.

Самым очевидным примером будет класс работы с датами:

```ts
+new Date(); // 1741963243818

await wait(5_000);

// Функция получения текущей даты будет возвращать разный результат в зависимости от времени запуска
+new Date(); // 1741963320257
```

Но `Date` не является единственным, к таковым можно отнести `localStorage`, `sessionStorage` и многие другие.

:::tip
Характерной чертой трудно тестируемых компонентов является наличие *внутреннего* неконтролируемого тестами состояния.
Для сервера - это может быть база данных, для web-api это состояние браузера или внутреннее состояние операционной
системы клиента.
:::

## Исключение периферии

Некоторые компоненты внешних (по отношению к web приложению) систем сложно протестировать. К примерам таковых можно
отнести:

* Использование нативных диалоговых окон
* Использование нативных функций устройств (например звуки, вибрация, сигналы)
* Использование периферийных устройств

Исходя из выбранного метода верификации, прямое тестирование данных компонентов не представляется возможным и должно
быть заменено на журналирование вызовов.
